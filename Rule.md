# 개발 규칙 (Rule.md)

이 문서는 본 프로젝트에서 Codex가 코드를 생성하거나 수정할 때 반드시 따라야 하는 개발 규칙을 정의한다.  
모든 코드는 **읽기 쉬움**, **직관적 구조**, **유지보수 용이성**, **실험 재현 가능성**을 최우선으로 한다.  
과도하게 복잡한 추상화, 불필요한 디자인 패턴, 이해하기 어려운 축약형 코드는 지양한다.

---

## 1. 코드 스타일 기본 원칙

### 1-1. 어려운 코드 금지
- 너무 어려운 코드, 과도한 추상화, 불필요한 클래스 분리, 복잡한 체이닝 코드는 만들지 않는다.
- 초보 개발자도 흐름을 따라갈 수 있도록 직관적으로 작성한다.
- 한 번에 이해되지 않는 로직은 단계별로 나누어 작성한다.
- "짧지만 이해하기 어려운 코드"보다 "조금 길어도 이해하기 쉬운 코드"를 우선한다.

### 1-2. 읽기 쉬운 이름 사용
- 변수명, 함수명, 클래스명은 역할이 바로 드러나게 작성한다.
- 축약어 사용을 최소화한다.
- 예: `q`, `res`, `tmp` 같은 이름보다 `user_question`, `query_result`, `parsed_schema` 같은 이름을 사용한다.

### 1-3. 함수는 한 가지 역할만 수행
- 하나의 함수는 하나의 책임만 갖도록 작성한다.
- 함수가 너무 길어지면 적절히 분리한다.
- 한 함수 안에서 입력 처리, 비즈니스 로직, 출력 포맷팅, DB 실행을 전부 처리하지 않는다.

### 1-4. 파일도 한 가지 역할 중심으로 작성
- 파일 하나에는 하나의 중심 역할만 둔다.
- 여러 역할이 섞이면 파일을 분리한다.
- 예: DB 연결, SQL 생성, SQL 검증, 벡터 검색, API 호출은 가능하면 별도 파일로 분리한다.

## 2. 주석 규칙

### 2-1. 파일 상단 한국어 주석 필수
- 모든 Python 파일 상단에는 해당 파일의 역할을 설명하는 3~5줄의 한국어 주석을 반드시 작성한다.
- 이 주석에는 아래 내용이 포함되어야 한다.
  1. 이 파일의 주요 목적
  2. 어떤 기능을 담당하는지
  3. 다른 파일과 어떻게 연결되는지

예시:
```python
# 이 파일은 사용자 질문을 기반으로 SQL을 생성하는 역할을 담당한다.
# LLM 또는 RAG 검색 결과를 바탕으로 프롬프트를 구성하고 SQL 초안을 만든다.
# 생성된 SQL은 별도의 validator 모듈에서 검증한 뒤 실행 단계로 전달된다.
# 가능한 한 직관적인 흐름으로 작성하며, 디버깅이 쉽도록 로그를 상세히 남긴다.
```

### 2-2. 함수 주석 작성
- 모든 주요 함수에는 한국어 docstring 또는 주석으로 역할을 설명한다.
- 아래 항목이 드러나도록 작성한다.
  - 함수 목적
  - 입력값
  - 반환값
  - 주의사항(있다면)

### 2-3. 주석은 "왜"를 설명
- 코드 한 줄을 그대로 번역하는 주석은 지양한다.
- 왜 이 로직이 필요한지, 어떤 의도로 작성했는지를 설명한다.

---

## 3. 로그 규칙

### 3-1. 로그 출력은 필수
- 주요 처리 단계마다 반드시 로그를 남긴다.
- 최소한 아래 단계에서는 로그를 기록한다.
  - 함수 시작
  - 주요 입력값 확인
  - 외부 호출 전/후
  - 예외 발생 시점
  - 함수 종료 또는 주요 결과 반환 시점

### 3-2. 로그 정보 형식
- 로그에는 반드시 아래 정보가 포함되어야 한다.
  - 시간
  - 로그 레벨
  - 파일명
  - 함수명
  - 라인 번호
  - 메시지

예시 형식:
```python
[2026-04-06 10:15:23] [INFO] [sql_generator.py:generate_sql:42] SQL 생성 시작
```

### 3-3. print보다 logging 우선
- 단순 `print()` 대신 Python `logging` 모듈을 우선 사용한다.
- 프로젝트 공통 logger 설정 파일을 두고 재사용 가능하게 구성한다.

### 3-4. 민감 정보 로그 금지
- 비밀번호, API Key, DB 계정 비밀번호, 토큰 원문은 로그에 남기지 않는다.
- 필요한 경우 마스킹 처리한다.

---

## 4. 예외 처리 규칙

### 4-1. 예외를 숨기지 않기
- `except Exception: pass` 같은 코드는 금지한다.
- 예외가 발생하면 원인을 로그로 남기고, 필요한 경우 다시 raise 한다.

### 4-2. 사용자에게 이해 가능한 메시지 제공
- 에러 메시지는 디버깅 가능한 수준으로 명확하게 작성한다.
- 단, 민감한 내부 정보는 노출하지 않는다.

### 4-3. 외부 시스템 호출은 반드시 보호
- DB 호출, API 호출, 파일 읽기/쓰기, 벡터DB 조회, LLM 호출은 반드시 예외 처리를 포함한다.

---

## 5. 구조 및 유지보수 규칙

### 5-1. 설정값 하드코딩 금지
- DB 접속 정보, API URL, 모델명, 경로, 임계값 등은 코드에 직접 하드코딩하지 않는다.
- 반드시 `config.yaml`, `.env`, 또는 별도 설정 파일에서 관리한다.

### 5-2. Notebook은 실험, 로직은 py 파일
- Jupyter Notebook에는 실험 흐름과 결과 확인만 작성한다.
- 핵심 비즈니스 로직은 반드시 `.py` 파일에 작성하고 Notebook에서는 이를 호출하는 방식으로 구성한다.

### 5-3. 공통 로직 재사용
- 같은 로직이 2번 이상 반복되면 공통 함수로 분리한다.
- 단, 과도한 공통화는 피하고 읽기 쉬움을 우선한다.

### 5-4. 파일/폴더명 일관성 유지
- 파일명과 폴더명은 역할이 바로 보이게 작성한다.
- 예: `sql_generator.py`, `db_executor.py`, `chroma_service.py`, `api_client.py`

---

## 6. 함수 작성 규칙

### 6-1. 타입 힌트 사용
- 가능한 모든 함수에 타입 힌트를 작성한다.
- 입력값과 반환값이 명확히 보이도록 한다.

### 6-2. 함수 길이 제한
- 함수는 가능하면 30~50줄 이내로 유지한다.
- 50줄을 넘기면 분리 가능성을 먼저 검토한다.

### 6-3. 입력과 반환 명확화
- 함수는 입력값과 반환값이 분명해야 한다.
- 여러 종류의 값을 애매하게 반환하지 않는다.

### 6-4. 전역 상태 최소화
- 전역 변수 사용을 최소화한다.
- 상태가 필요한 경우 클래스 또는 명시적 인자 전달 방식으로 처리한다.

---

## 7. Text-to-SQL / RAG 프로젝트 전용 규칙

### 7-1. SQL 안전성 우선
- 기본적으로 SELECT 조회 중심으로 작성한다.
- DELETE, UPDATE, INSERT, DROP, ALTER 등 데이터 변경 SQL은 사용자가 명시적으로 요청한 경우가 아니면 생성하지 않는다.

### 7-2. 생성된 SQL은 검증 후 실행
- LLM, RAG, API 어떤 방식으로 생성된 SQL이든 바로 실행하지 않는다.
- 별도의 검증 단계(validator)를 거친 뒤 실행한다.

### 7-3. 프롬프트는 파일로 분리
- 프롬프트 문자열을 코드 내부에 길게 하드코딩하지 않는다.
- `prompts/` 폴더의 텍스트 파일 또는 템플릿 파일로 분리한다.

### 7-4. RAG 검색 결과는 확인 가능해야 함
- RAG 기반 생성 시 어떤 문서를 검색했는지 확인 가능하도록 로그를 남긴다.
- 최소한 top-k 결과의 문서명 또는 chunk 요약을 남긴다.

### 7-5. 비교 실험 가능 구조 유지
- LLM 단독 / RAG 기반 / API 기반 결과를 같은 형식으로 비교할 수 있도록 통일된 반환 구조를 사용한다.

예시:
```python
{
    "method": "rag_sql",
    "question": "...",
    "generated_sql": "...",
    "validated": True,
    "execution_success": True,
    "error": None,
    "rows": [...]
}
```

## 8. 경로 및 파일 처리 규칙

### 8-1. pathlib 우선
- 파일 경로 처리는 문자열 결합보다 `pathlib.Path`를 우선 사용한다.

### 8-2. 인코딩 명시
- 파일 입출력 시 UTF-8 인코딩을 기본으로 사용한다.

### 8-3. 상대경로 기준 일관성 유지
- 프로젝트 루트를 기준으로 경로를 관리한다.
- Notebook, script, module 실행 위치가 달라도 동작하도록 작성한다.

---

## 9. Codex 응답 방식 규칙

Codex는 코드를 생성하거나 수정할 때 아래 원칙을 반드시 따른다.

- 먼저 전체 구조를 단순하게 설계한다.
- 한 번에 너무 많은 파일을 만들기보다 단계적으로 만든다.
- 각 파일의 역할이 명확히 보이게 작성한다.
- 생성한 코드에는 충분한 한국어 주석을 포함한다.
- 로깅이 빠지지 않도록 한다.
- 복잡한 패턴보다 직관적인 패턴을 선택한다.
- 추후 Notebook에서 바로 호출 가능한 구조로 작성한다.
- 코드를 생성하기 전에 항상 "이 코드가 초보 개발자도 바로 이해 가능한가?"를 먼저 판단하고, 그렇지 않다면 더 단순한 구조로 다시 작성한다.
- 항상 단순하고 유지보수 가능한 구조를 우선 선택하며, 복잡한 패턴이 필요하다면 왜 필요한지 주석으로 설명한다.

---

## 10. 금지 사항

- 불필요하게 어려운 추상화
- 이유 없는 클래스 남발
- 과도한 one-liner
- 민감정보 하드코딩
- 의미 없는 주석
- 예외 무시
- 로그 없는 핵심 처리
- 검증 없는 SQL 실행
- Notebook 안에 모든 로직을 몰아넣는 구조
